# Vibe Coding：从感觉到工程纪律

Vibe Coding 不是只凭感觉写代码。对我来说，它更像一种 AI 协作方式：先用直觉快速靠近方向，再用工程纪律把结果固定下来。

> 掘金首发：[https://juejin.cn/post/7655523471275376691](https://juejin.cn/post/7655523471275376691)

## 为什么需要这个概念

AI 让开发过程变快了，但也让很多判断变得隐形：为什么这样设计，为什么改这个文件，为什么相信这次构建可以交付。如果这些判断没有被记录，下一次接力就会重新消耗上下文。

所以 Vibe Coding 需要补上三个动作：

- 把模糊方向拆成可执行任务。
- 把 AI 输出放回构建、测试、浏览器和用户路径里验证。
- 把经验写回 `CLAUDE.md`、`LESSONS.md` 或项目文档，让下一次接力少走弯路。

## 我采用的工作流

第一步是快速成型。用 AI 帮我搭出界面、接口或功能骨架，先看到真实形状。

第二步是压实约束。检查目录结构、依赖、样式系统、验证命令和边界条件，避免只是“看起来能跑”。

第三步是复盘沉淀。把失败原因、版本陷阱、设计选择和可复用片段写成项目记忆。

## 最重要的分界线

Vibe Coding 的价值在速度和探索，但交付必须回到工程证据：代码能构建，页面能打开，交互能走通，改动能解释，后续的人或 AI 能接住。

这也是 kllin AI Lab 的核心主题：让 AI 应用可验证、可追踪、可评测、可运维。
